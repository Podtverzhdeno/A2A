# Спецификация MVP

## 1. Цель

Проверить на реальных B2B-закупках, что A3 Сбера способен безопасно получить потребность от A1, параллельно собрать предложения нескольких A2, объяснимо сравнить их и провести подтверждённую человеком сделку до заказа, платежа и документов.

## 2. Участники пилота

| Участник | Количество | Требование |
|---|---:|---|
| Покупатель | 1 | Клиент СберБизнес, определённый закупочный процесс |
| A1 | 1 | API или demo-адаптер ERP покупателя |
| Поставщики | 2–3 | Действующие договоры, стандартизированный каталог |
| A2 | 2–3 | Отдельный agent endpoint на поставщика |
| A3 | 1 | Платформа Сбера |
| ЭДО | 1 | Тестовый/пилотный контур |
| Пользователи | 3–10 | Закупщик, согласующий, администратор |

## 3. Товарная область

- одна категория стандартизированных MRO-товаров;
- 100–300 SKU;
- единицы измерения и обязательные атрибуты заранее согласованы;
- товары не требуют проектирования под заказ;
- отсутствуют специальные экспортные, медицинские или опасные ограничения;
- цена одной сделки и общий оборот ограничены правилами пилота.

## 4. Основной пользовательский сценарий

### US-01. Создание потребности

Как закупщик, я хочу передать товар, количество, адрес, срок и критерии выбора, чтобы A3 запросил предложения в пределах моего мандата.

Критерии:

- обязательные поля валидируются;
- потребность получает `deal_id`;
- бюджет не передаётся A2 по умолчанию;
- невалидный или истёкший мандат блокирует процесс.

### US-02. Параллельный RFQ

Как A3, я хочу отправить один нормализованный RFQ нескольким подходящим A2, чтобы получить конкурентные предложения.

Критерии:

- запрос отправляется минимум двум A2;
- каждому A2 передаётся одинаковое коммерческое ядро;
- оферты конкурентов изолированы;
- timeout одного A2 не блокирует остальных;
- повторная доставка RFQ не создаёт второй тендер у A2.

### US-03. Ответ поставщика

Как A2, я хочу вернуть подписанную оферту с ценой, наличием, сроком, гарантией и TTL.

Критерии:

- оферта соответствует схеме;
- указаны НДС, валюта, единица, количество и итог;
- указан срок действия;
- изменение существенных условий создаёт новую версию;
- просроченная оферта не может быть акцептована.

### US-04. Сравнение

Как закупщик, я хочу увидеть сопоставимые предложения и основания рейтинга.

Критерии:

- hard constraints отделены от score;
- для каждого предложения показаны цена, TCO, доставка, гарантия и риск;
- показаны веса критериев;
- показаны комиссии Сбера;
- результат можно воспроизвести по сохранённым входам и версии алгоритма;
- неизвестные характеристики явно помечены, а не заполнены моделью.

### US-05. Подтверждение

Как уполномоченный сотрудник, я хочу подтвердить снимок условий, чтобы выбор нельзя было незаметно изменить после моего решения.

Критерии:

- показан approval snapshot;
- snapshot имеет hash;
- проверяются полномочия и актуальность мандата;
- изменение цены, товара, количества, поставщика или срока аннулирует подтверждение;
- отказ закрывает или возвращает сделку на разрешённый этап.

### US-06. Заказ

Как A3, я хочу отправить award и создать один заказ у выбранного A2.

Критерии:

- заказ создаётся только после подтверждения;
- повтор команды возвращает исходный order ID;
- невыбранные A2 получают корректное завершение задачи;
- ответ поставщика записывается в Deal Ledger.

### US-07. Платёж

Как клиент, я хочу получить заполненный черновик платежа и подтвердить его штатным способом.

Критерии:

- A3 не подписывает платёж вместо клиента;
- сумма и получатель совпадают с approval snapshot;
- повтор запроса не создаёт второй платёж;
- authoritative payment status приходит из платёжного контура;
- расхождение создаёт exception.

### US-08. Документы и исполнение

Как клиент, я хочу видеть документы и статусы поставки в одной сделке.

Критерии:

- документы связаны с order/deal;
- хранится hash и источник;
- fulfillment events идемпотентны;
- нарушение SLA создаёт уведомление;
- закрытие возможно только после обязательных событий.

### US-09. Аудит

Как контролёр, я хочу восстановить последовательность сделки и причины решений.

Критерии:

- присутствуют actor, mandate и purpose;
- сохранены версии схем, policy и ranking;
- видны отправка, получение, retry и ошибки;
- доступ ограничен ролью;
- evidence bundle выгружается по `deal_id`.

## 5. Минимальный capability profile A2

```yaml
agent_role: supplier
capabilities:
  - rfq.receive
  - quote.create
  - quote.refresh
  - order.create
  - order.get
  - fulfillment.get
  - document.list
supported_categories:
  - mro.standardized
protocol_versions:
  - "1.0"
```

A2 не обязан использовать LLM. Корректный API-адаптер каталога поставщика является полноценным агентом для MVP, если поддерживает lifecycle, контракты и политики.

## 6. Минимальный набор сообщений

| Сообщение | Направление | Результат |
|---|---|---|
| `ProcurementIntent` | A1 → A3 | Создана сделка |
| `RFQ` | A3 → A2.N | Создана задача поставщика |
| `Quote` | A2.N → A3 | Зарегистрирована оферта |
| `Comparison` | A3 → A1 | Доступен рейтинг |
| `Approval` | A1 → A3 | Зафиксирован выбор |
| `Award` | A3 → selected A2 | Выбран поставщик |
| `OrderConfirmation` | A2 → A3 | Создан заказ |
| `PaymentIntent` | A3 → payment adapter | Создан черновик |
| `FulfillmentEvent` | A2 → A3 | Обновлён статус |
| `DealSummary` | A3 → A1 | Сделка закрыта |

## 7. Нефункциональные критерии MVP

### Безопасность

- mTLS между организациями;
- отдельные credentials каждого A2;
- scopes по операциям;
- отзыв доступа без перезапуска платформы;
- отсутствие секретов в логах;
- защита от replay;
- security negative tests перед live.

### Надёжность

- повтор входного сообщения безопасен;
- состояние сделки переживает перезапуск;
- один недоступный A2 не останавливает RFQ;
- есть reconciliation заказов и платежей;
- есть резервное копирование operational data и evidence.

### Производительность

Для пилота:

- p95 ответа gateway без внешних вызовов ≤ 1 секунда;
- создание fan-out задач ≤ 5 секунд;
- обновление статуса после callback ≤ 10 секунд;
- предел ожидания оферт задаётся бизнес-дедлайном, а не HTTP timeout.

### Аудит

- 100% изменяющих состояние действий имеют correlation ID;
- часы участников синхронизированы;
- критические события неизменяемы обычным пользователем;
- доступ к журналу фиксируется.

## 8. Definition of Done

Функция завершена, если:

- код прошёл review;
- есть unit и contract tests;
- обновлена схема;
- задокументированы ошибки;
- настроены метрики и алерты;
- проверены authorization и audit;
- нет критических уязвимостей;
- выполнен E2E на стенде;
- подготовлен rollback/runbook.

## 9. Критерии Go/No-Go в live

Go:

- все участники в allowlist;
- пройдены contract и security tests;
- установлены лимиты;
- включены подтверждения;
- работают kill switch и уведомления;
- дежурная команда имеет runbooks;
- reconciliation не показывает расхождений.

No-Go:

- A3 может создать платёж без штатного подтверждения;
- невозможно доказать, какой мандат разрешил действие;
- A2 получает данные конкурента;
- повтор сообщения создаёт второй заказ;
- решение ranking не воспроизводится;
- нет возможности немедленно отключить участника.

## 10. Демонстрационный сценарий приёмки

1. A1 запрашивает 20 единиц стандартизированного товара.
2. A3 проверяет мандат и находит три A2.
3. A2.1 отвечает быстро и дороже.
4. A2.2 отвечает дешевле, но нарушает требуемый срок.
5. A2.3 временно недоступен и отвечает после retry.
6. A3 исключает A2.2 по hard constraint и сравнивает A2.1/A2.3.
7. Пользователь видит причины и подтверждает A2.1.
8. Повторный `Approval` не создаёт второй order.
9. СберБизнес создаёт один черновик платежа.
10. Пользователь подтверждает платёж.
11. A2.1 передаёт статусы и документы.
12. Аудитор выгружает полный evidence bundle.

Сценарий считается пройденным, если данные не раскрылись конкурентам, все переходы воспроизводимы, а повторные сообщения не создали финансовых или операционных дублей.
